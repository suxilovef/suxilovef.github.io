01-单机部署与基础认知

1. 阶段定位

第一阶段是 Nginx 七阶段学习路线的入门基础阶段，目标不是记住大量配置项，而是先建立对 Nginx 的整体认知，掌握一台单机 Nginx 从安装、启动、配置、访问、日志查看到基础排障的完整闭环。

学习这一阶段时，需要把 Nginx 当成一个运行在操作系统上的网络服务来看，而不是只把它当成一份 nginx.conf 配置文件。你需要理解：

本阶段完成后，你应该具备下面能力：

  • 能在 Linux 服务器上独立安装并启动 Nginx。
  • 能说清 Nginx 是什么、适合做什么、不适合做什么。
  • 能理解 Nginx 的典型目录、配置文件、日志文件和运行用户。
  • 能读懂最小可运行的 nginx.conf。
  • 能配置一个基础静态站点并通过浏览器或 curl 访问。
  • 能使用 nginx -t、systemctl、nginx -s reload 完成基础管理。
  • 能排查启动失败、配置错误、端口占用、权限不足、远程访问失败等常见问题。

2. 学习边界

第一阶段只解决“单机部署与基础认知”，不要过早扩展到复杂代理、负载均衡、HTTPS、缓存、限流和源码。

2.1 本阶段重点

  • Nginx 的角色定位。
  • Nginx 与 Apache、Tomcat、Spring Boot 内置 Web 容器的区别。
  • 包管理器安装 Nginx。
  • 源码编译安装 Nginx。
  • Nginx 目录结构。
  • 主配置文件结构。
  • master/worker 进程模型。
  • systemd 服务管理。
  • 静态资源访问。
  • 基础日志查看。
  • 基础命令使用。
  • 常见入门故障排查。

2.2 本阶段暂不深入

下面内容后续阶段再系统学习：

  • server_name 的复杂匹配规则。
  • location 的优先级细节。
  • rewrite 和变量体系。
  • proxy_pass 反向代理细节。
  • upstream 负载均衡算法。
  • HTTPS、HTTP/2、证书续期。
  • gzip、brotli、缓存和性能调优。
  • 限流、限并发、访问控制。
  • 高可用、灰度发布、热升级。
  • Nginx 源码和模块开发。

3. Nginx 是什么

Nginx，读作 Engine X，是一个高性能的 HTTP Server、反向代理服务器、负载均衡器和通用 TCP/UDP 代理服务器。

在常见互联网架构中，Nginx 通常位于流量入口层：

Nginx 最常见的使用场景包括：

场景 说明
静态资源服务器 直接提供 HTML、CSS、JS、图片、字体、下载文件
反向代理 代理请求到后端 Tomcat、Spring Boot、Node.js、Go、PHP 等应用
负载均衡 将请求分发到多台后端服务
HTTPS 入口 统一处理 TLS 证书、HTTPS 握手、HTTP/2
API 网关前置层 做基础入口控制、Header 透传、访问日志、超时控制
缓存层 缓存静态文件或后端响应，降低后端压力
限流限并发 控制入口请求速率和连接数量
TCP/UDP 代理 使用 stream 模块代理 MySQL、Redis、TCP 服务等

第一阶段先重点掌握“静态资源服务器”和“单机 HTTP 服务”两个基础角色。

4. Nginx 的核心特点

4.1 高并发能力强

Nginx 采用事件驱动、异步非阻塞 IO 和 master/worker 多进程模型。相比一个连接一个线程的传统模型，Nginx 可以用较少进程处理大量并发连接。

4.3 资源占用低

Nginx 通常使用少量 worker 进程即可处理较高并发。相比传统应用服务器，它在静态资源、连接保持、请求转发等场景中资源消耗更低。

Nginx 由核心框架加模块组成。常见能力如 HTTP、SSL、gzip、rewrite、proxy、stream 等都来自模块。可以通过下面命令查看当前 Nginx 编译了哪些模块：

5. Nginx 与常见 Web 组件对比

5.1 Nginx 与 Apache

对比项 Nginx Apache HTTP Server
并发模型 事件驱动、异步非阻塞 多进程/多线程模型为主
静态资源性能 很强 较强
反向代理 常用且高效 支持，但现代入口层更常用 Nginx
配置风格 层级块配置 指令式配置，支持 .htaccess
内存占用 通常较低 并发高时可能更高
常见定位 入口代理、静态资源、负载均衡 传统 Web Server、兼容历史生态

5.2 Nginx 与 Tomcat

对比项 Nginx Tomcat
类型 Web Server / 反向代理 Java Servlet 容器
是否运行 Java Web 应用 不直接运行 直接运行 WAR/Servlet 应用
静态资源能力 强 一般
动态业务处理 不负责业务逻辑 负责 Java Web 业务逻辑
常见关系 位于 Tomcat 前面 位于 Nginx 后面

典型架构：用户 -> Nginx -> Tomcat / Spring Boot。

5.3 Nginx 与 Spring Boot 内置 Web 容器

Spring Boot 内置 Tomcat、Jetty 或 Undertow，可以直接监听端口提供 HTTP 服务。Nginx 通常放在 Spring Boot 前面，承担统一域名入口、HTTPS 证书卸载、静态资源服务、反向代理、访问日志、超时控制以及后续负载均衡和限流职责。Spring Boot 则负责业务接口、业务逻辑、数据库访问、权限认证和事务处理。

11.1 main 全局块

main 不是显式写出来的块，而是配置文件最外层上下文。

指令 示例 说明

events 块定义连接处理相关参数。

指令 说明

11.3 http 块

http 块是 HTTP 服务的核心上下文，绝大多数 Web 配置都写在这里或其子块中。

server 块表示一个虚拟主机，可以理解成一个站点或一个监听入口。

指令 说明

root 静态资源根目录

11.5 location 块

location 块用于匹配 URI 路径，并决定如何处理请求。

请求 GET /index.html，如果 root 是 /usr/share/nginx/html，最终查找文件就是 /usr/share/nginx/html/index.html。

12.1 user

user 指定 worker 进程使用哪个系统用户运行。master 进程通常由 root 启动，worker 会切换到普通用户以降低安全风险。

查看进程用户：

12.2 worker_processes

配置 说明
worker_processes 1; 一个 worker，适合学习实验
worker_processes auto; 自动按 CPU 核心数设置，生产常见
worker_processes 4; 手动指定 4 个 worker

入门建议使用 auto。

12.3 worker_connections

表示每个 worker 最大连接数。注意它不是整个 Nginx 的总连接数，而是单个 worker 的连接上限。

12.4 pid

pid 文件记录 master 进程 ID。执行 reload、stop、quit 时，Nginx 会根据 pid 找到 master 进程并发送信号。

12.5 error_log

错误日志用于记录启动错误、配置问题、权限问题、连接失败、上游错误等信息。

级别 说明
debug 调试级别，需要 debug 编译支持
info 普通信息
notice 重要通知
warn 警告，常用默认级别
error 错误
crit 严重错误
alert 需要立即处理
emerg 系统不可用级别

入门阶段建议使用 warn 或 error。

12.6 access_log

访问日志记录每一次 HTTP 请求的访问信息，例如客户端 IP、请求时间、URI、状态码、响应大小等。

12.7 include

include 用于引入其他配置文件，核心作用是拆分配置，避免所有内容都堆在一个 nginx.conf 中。

注意：被 include 的文件内容会被放入当前位置解析；子配置必须放在正确上下文中，例如 server 应该放在 http 内。

12.8 root

root 表示静态资源根目录。Nginx 会将 root 与 URI 拼接成真实文件路径。

12.9 index

index 表示访问目录时默认查找的首页文件。例如访问 http://example.com/，Nginx 会尝试查找 root目录/index.html 和 root目录/index.htm。

13. master/worker 进程模型

Nginx 采用 master/worker 多进程模型。

13.1 master 进程职责

master 进程主要负责管理，不直接处理普通 HTTP 请求。

职责包括：

  • 读取和校验配置。
  • 绑定监听端口。
  • 创建 worker 进程。
  • 接收 reload、stop、quit 等信号。
  • 管理 worker 生命周期。
  • 平滑重载配置。
  • 处理日志重新打开。

13.2 worker 进程职责

worker 进程负责实际处理客户端连接和 HTTP 请求。

职责包括：

  • 接收客户端连接。
  • 读取 HTTP 请求。
  • 匹配 server/location。
  • 读取静态文件。
  • 返回 HTTP 响应。
  • 写入访问日志。
  • 后续阶段还会负责代理 upstream。

14.2 reload 与 restart 区别

stop 停止服务 否 停机维护
start 启动服务 - 服务未运行时

19. HTTP 基础认知

Nginx 是 HTTP Server，因此需要掌握最基础的 HTTP 概念。

19.1 请求报文

一个 HTTP 请求大致包含请求行、请求头、空行和请求体。

19.2 响应报文

一个 HTTP 响应大致包含状态行、响应头、空行和响应体。

19.3 常见状态码

状态码 含义 入门阶段常见原因
200 成功 文件存在并正常返回
301/302 重定向 配置了 return/rewrite 或目录补 /
403 禁止访问 权限不足、目录无 index、禁止访问
404 未找到 URI 对应文件不存在，root 路径拼错
500 服务端内部错误 配置或后端异常，第一阶段较少见
502 Bad Gateway 代理后端失败，后续阶段重点
503 服务不可用 服务过载或人为返回
504 Gateway Timeout 代理后端超时，后续阶段重点

29. 本阶段常见误区

  • 只会执行 apt install nginx，不知道配置文件在哪里。
  • 修改配置后不执行 nginx -t。
  • 把 reload 和 restart 混为一谈。
  • 不看 error.log，只看浏览器报错页面。
  • 看到 403 就以为 Nginx 坏了，忽略文件权限和 index 文件。
  • 看到 404 就以为服务没启动，忽略 root 路径拼接。
  • 不区分包管理器安装和源码安装路径。
  • 同时启动两个 Nginx，导致端口冲突。
  • 云服务器远程访问失败时，只检查 Nginx，不检查安全组。
  • 使用 root 用户随意运行服务，不关注 worker 运行用户。
